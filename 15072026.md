## 1. SYSTEM OVERVIEW

SURAKSHA AI is an intelligent conversational crime analytics platform for the **Karnataka State Police**, deployed on **Zoho Catalyst**. It enables officers to query a **26-table FIR database** using natural language (English + limited Kannada), visualize criminal networks, detect crime hotspots, forecast trends, and generate investigation reports.

### 1.1 High-Level Architecture

```
                        Catalyst API Gateway
                     (POST /api/ -> suraksha_ai function)
                                  |
                    main.py -- Request Router
        72 actions dispatched by `action` parameter in JSON body
        RBAC gate -> Audit log -> Handler -> Response
          |          |          |          |
          v          v          v          v
     +------+ +----------+ +--------+ +--------------+
     | Chat | |Investi-  | |Analytics| |Comm-   | |Governance+ |
     | NL   | |gation    | |MOCK    | |ander   | |Comms       |
     |->ZCQL| |Suite     | |random.*| |Multi-  | |In-memory   |
     |2-path| |In-mem    | |        | |Agent   | |            |
     +------+ +----------+ +--------+ +--------------+
        |         |                        |
        v         v                        v
     +----------------------------------------------+
     |         Catalyst Data Store (26 tables)       |
     |  CaseMaster, Accused, Victim, Unit, ...       |
     |  ZCQL -- restricted SQL dialect, no JOINs      |
     +----------------------------------------------+
```

## 5. LIMITATIONS

### 5.1 ZCQL Constraints (Platform)

| Limitation | Impact |
|------------|--------|
| No JOINs | All cross-table queries need multi-step lookups (2-4 round-trips) |
| No LIKE | No partial/fuzzy text search on BriefFacts, AccusedName, etc. |
| No subqueries | Can't do `WHERE x IN (SELECT ...)` -- must pre-fetch IDs |
| No COUNT(\*) | Must use `COUNT(ROWID)` -- returns expression name, not alias |
| No functions | No LOWER(), YEAR(), UPPER() -- case-sensitive exact match only |
| No ORDER BY alias | Can't sort by aggregate expressions |
| No HAVING | Can't filter grouped results |
| No UNION/CTEs | No set operations |

### 5.2 Implementation Gaps

| Gap | Impact | Root Cause |
|-----|--------|------------|
| JOINs in production code | OffenderProfiler fails | `offender_profiler.py` uses JOIN syntax |
| Wrong table name | LeadGenerator fails | `lead_generator.py` queries `FirMaster` not `CaseMaster` |
| In-memory governance | Data loss on cold start | `governance.py` stores in dicts |
| In-memory investigations | Data loss on cold start | `investigation_manager.py` stores in dicts |
| Mock JWT | Not production auth | Custom HMAC, no OIDC, no MFA |
| N+1 inserts | Slow bulk operations | `datastore_client.py` one INSERT per row |
| No caching | Repeated DB hits | `common/cache/` empty |
| No health endpoint | No monitoring | `common/health/` empty |
| No frontend tests | Regression risk | Zero frontend test files |
| Frontend mock catch-all | Silenced errors | `api.ts` try/catch -> fallback data on any failure |

## 6. WHAT IS STOPPING END-TO-END SUCCESS

### 6.1 CRITICAL BLOCKERS

1. **ZCQL JOIN syntax in production code** -- `offender_profiler.py` and `lead_generator.py` contain explicit JOINs. ZCQL does not support JOINs. These will throw SQL parse errors in production.

2. **`lead_generator.py` references `FirMaster`** (line 85) -- table does not exist. The correct table is `CaseMaster`. This query will always fail with `Table not found`.

3. **All analytics modules are `random.*` mocks** -- TrendAnalyzer, HotspotDetector, Forecaster, AlertEngine, StatsAggregator, GraphProjector all return fake data. Every chart in the dashboard shows fabricated numbers.

4. **No entity resolution** -- `EntityResolver` is `class EntityResolver: pass`. Repeat offender detection, criminal network building, and offender profiling are impossible without it.

5. **In-memory persistence loss** -- InvestigationManager, ModelRegistry, PromptRegistry, MissionTracker, ClaimLedger are all in-memory dicts. Lost on every Catalyst cold start.

### 6.2 HIGH-RISK ISSUES

6. **Frontend try/catch swallows all backend errors** -- Every API call falls back to client-side mock data on ANY failure. The dashboard will appear to work while backend is completely down. Users see fabricated data with no indication of error.

7. **Kannada support is token** -- 2 greeting strings + 9 glossary terms. Full translation engine is absent despite the KSP requirement.

8. **No PDF generation** -- `generate_investigation_report` and `export_pdf` are stubs. No real document is produced.

### 6.3 DEPLOYMENT RISKS

9. **GLM model availability** -- Depends on `crm-di-glm47b_30b_it` being available in the Catalyst region. If deprecated or rate-limited, the NL query pipeline degrades to regex-only.

10. **OAuth token expiration** -- QuickML `Zoho-oauthtoken` retrieved at startup with no refresh mechanism. Long sessions will lose LLM capability.

11. **Missing Catalyst integrations** -- Voice (Zia STT), Cache, Stratus (file storage) are empty stubs. Features degrade silently.
